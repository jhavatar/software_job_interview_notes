Creational Design Patterns

Creational Design Patterns

Deal with object creation mechanisms, Create objects in a manner suitable to the situation.

Factory Method

Intent

  • Define a class for creating an object, but let subclasses decide which class to instantiate.
  • Standardize the architectural model for a range of applications, but allow for individual applications to define their own domain objects and provide for their instantiation.

Description

  • Instead of factory with method for each object type, define a factory class with a factory method.
  • Inheritance is used to create object type -- factory subclass overrides factory method.

Notes

  • -

Example

  • Manufacturers of plastic toys process plastic molding powder, and inject the plastic into molds of the desired shapes. The class of toy (car, action figure, etc.) is determined by the mold.
  • When staying in a hotel you first have to check in. The person working at the front desk will give you a key to your room after you've paid for the room you want and this way he can be looked at as a room factory. While staying at the hotel, you might need to make a phone call, so you call the front desk and the person there will connect you with the number you need, becoming a phone-call factory, because he controls the access to calls, too.

Abstract Factory

Intent

  • Provide an interface for creating families of related or dependent objects without specifying their concrete classes.
  • A way to encapsulate a group of individual factories that have a common theme without specifying their concrete classes

Description

  • A Factory interface is defined with multiple factory methods. A concrete factory that implements the interface is passed to another object to delegate object instantiation.
  • An Abstract Factory is a class with several Factory methods. So basically, an Abstract Factory class has a collection of different Factory Methods to create different desired Concrete objects.
  • Each concrete factory sub-class implements the Factory methods for a particular family of products. The Abstract Factory pattern can be considered as an extension of the Factory Method pattern.

Builder

Intent

  • Separate the construction of a complex object from its representation so that the same construction process can create different representations.
  • Encapsulate the construction of a product and allow it to be constructed in steps

Description

  • Components/Roles
    • Builder -- interface
    • Concrete Builder -- implementation of the Builder
    • Director -- invokes “builder” services on the Builder step by step as it interprets an external format
  • The Builder creates part of the complex object each time it is called and maintains all intermediate state
  • When the product is finished, the client retrieves the result from the Builder
  • Pros
    • Encapsulates the way complex object is created
    • Allows objects to be created in a multistep varying process
    • Hides internal representation of product from client
    • Product implementations can be swapped because client only sees an abstract interface
  • Cons
    • Constructing object requires more domain knowledge of the client than when using factory

Notes

  • -

Example

  • Vehicle Manufacturer: From the same of parts and construction steps, can build a car, a bicycle, a motorcycle or a scooter. In this case the Builder will become the VehicleBuilder. The ConcreteBuilders are CarBuilder, MotorCycleBuilder, ...

Object Pool

AKA Resource Pool

Intent

  • improve managing object caching by avoiding creating instances by reusing them

Description

  • A client with access to a Object pool can avoid creating a new Objects by simply asking the pool for one that has already been instantiated instead.
  • The pool can create new objects if empty, or restrict the number of objects created.
  • Effective in situations:
    • cost of initializing is high
    • the rate of instantiation is high
    • number of instantiations in use at any one time is low

Notes

  • usually implemented as Singletons
  • Factory Method pattern can be used to encapsulate the creation logic for objects

Example

  • Database connections. Opening connection each time affects performance. Can also limit the maximum number of connections possibel at a time.

Prototype

Intent

  • Avoid the cost of creating a new object in the standard way (e.g., using the 'new' keyword) when it is prohibitively expensive for a given application.
  • Avoid subclasses of an object creator in the client application,

Description

  • Specify the kinds of objects to create using a prototypical instance, and create new objects by copying this prototype.
  • Implements clone() operation
  • Use one instance of a class for use as a breeder of all future instances.

Notes

  • -

Example

  • The mitotic division of a cell - resulting in two identical cells
  • A sales analysis on a set of data from a database. We copy the information from the database, encapsulate it into an object and do the analysis. But if another analysis is needed, it is easier just to clone then the object used in the first analysis for the other analysis.

Singleton

Intent

  • Require a single instance
  • provide global point of access to it
  • Encapsulate lazy initialization

Description

  • The single instance is a private static attribute.
  • The accessor function is a public static method.
  • Use one instance of a class for use as a breeder of all future instances.

Notes

  • -

Example

  • Logger classes. Provides a global logging access point in all the application components without being necessary to create an object each time a logging operations is performed.
