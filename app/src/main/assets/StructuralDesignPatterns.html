Structural Design Patterns

Structural Design Patterns

Design patterns that identifying a simple way to realize relationships between entities

Adapter

Intent

  • Make two incompatible interfaces to work together.
  • Convert the interface of a class into another interface clients expect

Description

  • components:
    • Adaptee class: legacy class
    • Target Interface class: interface expected by client
    • Adapter object: make adaptee available via target interface
  • Does not modify adaptee class
  • 2 Types:
    • Object adapter, aka Wrapper:
      • Makes use of composition
      • Adapter object implements interface class and contains adaptee object
    • Class adapter:
      • makes use of inheritance
      • extends both target interface class and legacy adaptee class
      • requires multiple inheritance

Notes

  • Class adapter's multiple inheritance makes it unusable on so,e languages, e.g. Java
  • Pattern gets retrofitted, makes things work after they’re designed

Bridge

Intent

  • Decouple an abstraction from its implementation so that the two can vary independently
  • run-time binding of the implementation

Description

  • components:
    • Handle: interface object known and used by client
    • Body: implementation object that is encapsulated by handle
  • Each component has its own hierarchy and can be extended independently

Notes

  • makes things work before they’re designed

Example

  • A switch that switches a device on and off -- switch can be chain or flip-switch, device can be light or TV

Composite Pattern

Intent

  • Clients treat individual objects and compositions of objects uniformly.
  • Compose objects into a tree structures to represent whole-part hierarchies.
  • Support recursive composition -- composites that contain components, each of which could be a composite

Description

  • components:
    • primitive object
    • composite object -- built from combining primitive or other composite objects
    • Component: abstract base class that specifies the behavior that needs to be exercised uniformly across all primitive and composite objects

Notes

  • -

Example

  • Arithmetic expression consisting of an operand, an operator (+ - * /) and another operand -- operand can be number or another arithmetic expresssion. e.g. 2 + 3 and (2 + 3) + (4 * 6)

Decorator

Intent

  • Attach additional responsibilities to an object dynamically -- e.g. add behavior and state to object at run-time
  • Flexible alternative to subclassing for extending functionality.

Description

  • components:
    • single abstract class or interface
    • core object
    • decorator objects
  • decorators implement same interface or abstract class as core object
  • decorator adds new methods or extends core's methods
  • using inheritance to achieve type matching, not to get behavior.

Notes

  • Can implement new decorator at any time to add new behaviour -- do not require change existing code

Example

  • Coffee shop Advertises and prices its products using abstract class Beverage. Create variations of base class coffee by adding delegates like Milk, Whip, Mocha, ...

Facade

Intent

  • unified interface to a set of interfaces in a subsystem
  • defines a higher-level interface that makes the subsystem easier to use

Description

  • object that provides a simplified interface to a larger body of code (such as a class library)
  • facade features:
    • code easier to use, understand and test -- convenient methods for common tasks
    • more readable
    • reduce dependencies of outside code on the inner workings
    • wrap a poorly designed collection of APIs with a single well-designed API

Notes

  • -

Example

  • class library
  • Home theater system -- hides amplifier, dvd player, projector, screen, tuner

Flyweight

Intent

  • object that minimizes memory use by sharing as much data as possible with other similar objects
  • way to use objects in large numbers when a simple repeated representation would use an unacceptable amount of memory

Description

  • Each flyweight object is divided into two pieces:
    • state-dependent (extrinsic) part stored (shared) in the Flyweight object
    • state-independent (intrinsic) part stored or computed by client objects, and passed to the Flyweight when its operations are invoked.

Notes

  • -

Example

  • graphical representation of characters in a word processor -- a glyph object for each character would amount to hundreds or thousands of bytes, Instead, for every character a reference to a flyweight glyph object shared by every instance of the same character in the document; only the position of each character (in the document and/or the page) would need to be stored internally

Private Class Data

AKA: PIMPL (Private IMPLementation) or Opaque pointer

Intent

  • seeks to reduce exposure of attributes by limiting their visibility.
  • Control write access to class attributes
  • Separate data from methods that use it
  • Encapsulate class data initialization

Description

  • Components:
    • data class -- move all attributes that need hiding to the data class
    • main class -- encapsulate data class object.
  • main class object initializes data class object in constructor.
  • main class object exposes attributes of data class through a getter.
  • expose attributes that will change through a setter.

Proxy

Intent

  • Provide a surrogate or placeholder for another object to control access to it.
  • Use an extra level of indirection to support distributed, controlled, or intelligent access.
  • Add a wrapper and delegation to protect the real component from undue complexity.

Description

  • proxy object instantiates the real object the first time the client makes a request of the proxy

Notes

  • situations where applicable
    • placeholder for “expensive to create” objects
    • provides a local representative for an object that resides in a different address space
    • controls access to a sensitive master object
    • performs additional actions when an object is accessed.

Example

  • A check or bank draft is a proxy for funds in an account. A check can be used in place of cash for making purchases and ultimately controls access to cash in the issuer’s account.
