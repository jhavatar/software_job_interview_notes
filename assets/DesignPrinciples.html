Design Principles

Object-oriented Design Principles

Abstraction principle

Encapsulate what varies

  • Identify the aspects of your application that vary and separate them from what stays the same.
  • result is fewer unintended consequences from code changes and more flexibility in your system

Program to interface, not implementation

  • Exploit polymorphism by programming to a a supertype so that the actual runtime object isn't locked into code.
  • Note, the concept of an interface, i.e. supertype -- not necessarily a Java interface.

Composite Reuse Principle

Favour composition over inheritance

  • HAS-A relationship over IS-A
  • Creating a system using composition gives you flexibility, e.g. change behaviour at runtime

Strive for loosely coupled designs between objects that interact

  • When 2 objects are loosely coupled they can interact but have very little knowledge of each other
  • Allows building flexible OO systems that can handle change because they minimize the interdependency.

Open/Closed Principle

Classes should be open for extension but closed for modification

  • should be that an entity (classes, modules, functions, etc.) can allow its behaviour to be modified without altering its source code
  • The closed parts should be isolated from new extensions

Dependency inversion principle

Depend on abstractions. Do not depend on concrete classes

  • Reduce dependencies
  • High-level components should not depend on our low-level components. Both should depend on abstractions.
  • Guidelines:
    • No variable should hold reference to a concrete class. Can use factory to get around "new".
    • No class should derive from a concrete class -- if derive from concrete class then depend on concrete class.
    • No method should override an implemented method of any of its base classes -- if override then base class not abstract. Implemented methods should be shared by all subclasses.

Principle of least knowledge

only talk to your friends

  • Reduce interactions between objects to just a few close friends.
  • Only invoke methods that belong to:
    • Object itself
    • Objects passed in as parameters to a method
    • Any object method created or instantiates
    • Any components of the object

Hollywood Principle

Don't call us, we'll call you

  • Prevents dependency rot.
  • Allow low-level components to hook themselves into system, but high-level components determine when they are needed and how.

Single responsibility principle

A class should have only one reason to change

  • Every class should have a single responsibility, and that responsibility should be entirely encapsulated by the class
  • All its services should be narrowly aligned with that responsibility.
